## ความน่าเชื่อถือของข้อมูล (Data Integrity)

> **บทเรียนจากโครงการสวดลักขี:** เคยพบปัญหาผู้ใช้บันทึกยอดสูงผิดปกติ เนื่องจากไม่ได้ตีกรอบการบันทึกไว้ดีพอ

**ความกังวลหลัก — Anti-fraud / Anti-spam** ต้องป้องกันการปั๊มยอดนาทีเพื่อให้ข้อมูลมีความศักดิ์สิทธิ์และน่าเชื่อถือ

| กลไกควบคุม | รายละเอียด |
|-----------|-----------|
| จำกัดนาทีต่อวันต่อคน | กำหนดเพดานที่สมเหตุสมผลกับการปฏิบัติวิทิสาจริง |
| ผ่านสาขาตรวจสอบ | การบันทึกรายบุคคลต้องอยู่ภายใต้การดูแลของสาขา |
| ล็อคด้วยวิธีการปฏิบัติ | ระบบผูกกับรูปแบบการทำวิทิสา มีเวลาที่สมเหตุสมผลต่อครั้ง |
| Anti-Spam Filter | ตรวจจับความผิดปกติอัตโนมัติ เช่น User เดียวบันทึก 24 ชม.ต่อเนื่อง |

## ความต้องการด้าน Infrastructure

ระบบต้องรองรับผู้ใช้งานจำนวนมากจากทั่วประเทศพร้อมกัน โดยใช้แนวทางจาก [Infrastructure Checklist](https://github.com/monthop-gmail/ai-skill/blob/master/infrastructure/dev-env/CHECKLIST.md) ครอบคลุม:

1. **Local Dev Architecture** — 1 service = 1 repo, Docker Compose, CF Tunnel
2. **Local to Production Sync** — compose เหมือนกันทุก env, ต่างกันแค่ `.env`
3. **Environment Separation** — DNS แยก env, git branching, branch protection, CI/CD
4. **Secret Management** — SOPS + age encryption
5. **Overall Structure** — directory layout, modular multi-service
